Stable IDs: All You Need to Know

Stable IDs are IDs that you set yourself as opposed to IDs that are generated by OpenUI5. Stable means that the views and view IDs are completely set by you as an application developer, the IDs are concatenated with the application component ID or do not have auto-generated parts.

Caution

We strongly suggest that you use stable IDs whenever possible, since they are an important prerequisite for SAPUI5 flexibility, automated testing, and inline help tools (such as Web Assistant). Apps with missing stable IDs are of worse quality and offer less functionality to customers. However, some technical controls don't need stable IDs (see note below).

Background: Why Are Stable IDs Needed?

By default, OpenUI5 generates IDs for instantiated controls. For example, the page and the table in the following XML view could have the generated IDs __page0 and __table0 at runtime.

<mvc:View
   xmlns="sap.m"
   xmlns:mvc="sap.ui.core.mvc">
   <Page>
      <content>
         <Table>
         </Table>
      </content>
   </Page>
</mvc:View>

These generated IDs change whenever the control structure of the app changes. They are also influenced during instantiation. If there are two views with unstable IDs in the app, depending on the order the views are opened, each view could get the generated ID __view0 or __view1.

This is an issue for the following features that require stable IDs:

  • SAPUI5 flexibility

    Allows you to extend apps based on your requirements, for example, by creating variants or adapting the user interface at runtime. Stable IDs are used to identify the controls that are to be adapted.

  • Automated tests

    To check the behavior of apps at runtime, these tests find controls by searching for stable IDs. The amount of stable IDs required depends on the respective framework. If you use OPA in OpenUI5, you're able to find controls via other criteria like control type, display name and others. For more information, see Integration Testing with One Page Acceptance Tests (OPA5).

  • Inline help tools

    These tools display user assistance information directly in the app and depend on stable IDs (example: Web Assistant).

The solution for this is that you as an application developer set the IDs manually to keep them stable. For example, if you set the ID of the following XML view, the contained page and table would have the IDs myView--page and myView--table at runtime.

<mvc:View
   xmlns="sap.m"
   xmlns:mvc="sap.ui.core.mvc">
   <Page id="page">
      <content>
         <Table id="table">
         </Table>
      </content>
   </Page>
</mvc:View>

The XML view prefixes the control IDs with its own ID. This allows you to use the same control ID for different views and to use the same view multiple times. For more information, see Support for Unique IDs.

For JavaScript views, JavaScript-generated controls, and XML-based views, you have to use the createID method of the view or control for this purpose.

Caution

If after a software upgrade some controls disappeared or the way in which they can be identified changed, this has a direct impact on the features relying on stable IDs. For this reason, the IDs must be kept stable over the lifecycle of the app; they are part of the public API of the app.

For more information, see XML View.

How to Use Stable IDs

The standard use case is that you use stable IDs for the view that the router navigates to. Ideally, you create your views with routing targets and declare the view ID in the manifest instead of setting it yourself in the view (as shown in the example below). For more information, see Routing and Navigation and Descriptor for Applications, Components, and Libraries.

{
   "sap.ui5": {
      "routing": {
         "targets": {
            "myTarget": {
               "viewName": "MyView",
               "viewId": "myView"
            }
         }
      }
   }
}

If you do not use the standard use case, you embed your view by setting its ID (the ID for the enclosing view is generated as in the previous example). Here's an example:

<mvc:View xmlns="sap.m"
xmlns:mvc="sap.ui.core.mvc">
<Page id="page">
<mvc:XMLView id="myView" viewName="MyView" />
</Page>
</mvc:View>
Note

Bear in mind that some technical controls, such as CustomData or aggregation binding templates, don't need stable IDs. Also do not set stable IDs for controls in bound aggregations because it's not actually possible. For example, the items in the following table are generated along the bound data and therefore cannot have stable IDs.

<mvc:View
   xmlns="sap.m"
   xmlns:mvc="sap.ui.core.mvc">
   <Table
      id="table"
      items="{/ProductCollection}">
      <items>
         <!-- no ID is set for controls in the bound 'items' aggregation -->
         <ColumnListItem>
            <!--template elements -->
         </ColumnListItem>
      </items>
   </Table>
</mvc:View>
How to Name Stable IDs

When giving names to stable IDs, choose IDs that describe the semantics of your views and controls, such as page or table. For example, if you're building an app using the SAP Fiori Worklist Application template, the following concatenated IDs are generated at runtime. For more information, see Worklist Template.

  • myProducts---worklist--page
  • myProducts---worklist--table
  • myProducts---product--page
  • myProducts---product--objectHeader
